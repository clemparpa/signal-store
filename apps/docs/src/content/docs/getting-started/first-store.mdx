title: Your first store
description: Build a counter store with withState, withComputed, withMethods, and patchState.
---

A store is composed by passing features to `signalStore`. The most common features are `withState` (declare initial state), `withComputed` (derived values), and `withMethods` (mutations + actions).

## A counter store

```ts
import {
	signalStore,
	withState,
	withComputed,
	withMethods,
	patchState,
} from '@fluch/signal-store';
import { computed } from '@preact/signals-core';

export const counter = signalStore(
	withState({ count: 0 }),

	withComputed(({ count }) => ({
		double: computed(() => count.value * 2),
		isPositive: computed(() => count.value > 0),
	})),

	withMethods((store) => ({
		increment: () => patchState(store, { count: store.count.value + 1 }),
		decrement: () => patchState(store, { count: store.count.value - 1 }),
		reset: () => patchState(store, { count: 0 }),
	})),
);
```

## Reading state

Each top-level key of the state becomes its own signal. Read it via `.value`:

```ts
counter.count.value; // 0
counter.double.value; // 0
counter.isPositive.value; // false
```

## Calling methods

```ts
counter.increment();
counter.increment();
counter.count.value; // 2
counter.double.value; // 4
counter.isPositive.value; // true
```

## Reactivity

In React (with `@preact/signals-react` installed in the app), reading `counter.count.value` inside a component subscribes that component to changes — no `useState`, no selector, no memoization needed. A call to `counter.increment()` re-renders only the components that read `count` or any computed derived from it.

## Lifecycle hooks

To run code at store construction or teardown, compose `withHooks({ onInit, onDestroy })`:

```ts
import { signalStore, withState, withHooks, patchState } from '@fluch/signal-store';

const counter = signalStore(
	withState({ count: 0 }),
	withHooks({
		onInit(s) {
			// runs once, after every feature is composed
			patchState(s, { count: 1 });
		},
		onDestroy(s) {
			console.log('final count:', s.count.value);
		},
	}),
);

counter.count.value; // 1
```

- **`onInit`** fires synchronously after `signalStore(...)` finishes composing every feature. It can read every signal/computed and call `patchState` — useful for seeding the store from a one-time source.
- **`onDestroy`** fires when `destroyStore(store)` runs (also automatically when a React `<Provider>` unmounts). The store is still **live** at that point: `patchState` propagates, signals are readable.

Multiple `withHooks(...)` compose: `onInit` callbacks run in composition order, `onDestroy` in reverse (LIFO).

Hooks are synchronous. For fire-and-forget async work, wrap in an explicit IIFE so the intent is visible:

```ts
withHooks({
	onInit(s) {
		void (async () => {
			const data = await fetchInitial();
			patchState(s, { data });
		})();
	},
});
```

For cancellable async pipelines (debounce, switchMap, error recovery), `rxMethod` is the right tool — see the [rxMethod guide](/signal-store/guides/rx-method/).

## Cleanup

Stores set up internal subscriptions. When a store is no longer needed (e.g. it was scoped to a component lifecycle), call `destroyStore` to release them:

```ts
import { destroyStore } from '@fluch/signal-store';

destroyStore(counter);
```

`destroyStore` is idempotent and safe to call multiple times. After it runs, further `patchState` calls on that store are silently dropped.
